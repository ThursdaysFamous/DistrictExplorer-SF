# Mechanization Playbook — Converting Convention into Machinery

**ARCHIVED 2026-07-13 — all three conversions DONE, all three drills recorded, all three
demotions fired. The metro-#3 gate is open. Archived per this document's own self-archival
clause; the machinery it built is documented where it runs (ENGINE_SYNC.md, the workflows,
the generator).**

Status: completed. Scope: three conversions only. Owner: CHI (reference implementation).
Cross-refs: docs/ENGINE_SYNC.md, docs/METRO_EXPANSION_PLAYBOOK.md,
docs/OPTIMIZATION_PLAYBOOK.md, docs/REDISTRICTING_RUNBOOK.md.

## Preamble

Everything machine-enforced in this project is healthy; everything convention-enforced has
drifted at N=2 metros. NYC's CLAUDE.md is a stale Chicago copy (wrong city, wrong geocoder,
wrong workflows, wrong ground-truth point). NYC's docs/BUILD_PLAYBOOK_1.md and
docs/OPTIMIZATION_PLAYBOOK.md are verbatim stale Chicago copies. The pointer-stub pattern was
applied to only one of three duplicated playbooks. Validator improvements (EXPECT_LAYER_IDS,
LAYER_AREA_RANK) were born in the NYC fork and CHI still lacks them. The engine loaders
loadArcGISPaged and makeSocrataPointLoader were born in NYC and back-ported by hand. That is the
whole evidence base: at two metros the conventions already failed. At the planned metro count
they will fail worse. This document converts exactly three conventions into machinery.

The anti-pattern this document must NOT become: in this project, playbooks become the
deliverable. The 371-line OPTIMIZATION_PLAYBOOK.md was written and its back-port never happened;
doc drift occurred while the doc about doc drift was being written. A playbook that is merely
well-written has changed nothing.

Standing rule — the definition of done for every conversion here: a conversion is DONE only when
its check has FAILED at least once in CI or a scripted drill, i.e. the machine has demonstrably
caught the thing the convention used to be responsible for. Until first red, the conversion is
"in progress," the prose remains authoritative, and no new metro may be provisioned. Each section
below records this in one line: "Prose demoted when: …".

Recommended execution order: **1 → 2 → 3.** Conversion 1 (engine artifact) makes parity true by
construction and unblocks everything else; without it, doc generation still leaves the engine
hand-ported. Conversion 2 depends on the worksheet schema that Conversion 1's assembly step
already needs to read metro config. Conversion 3 (reverse-parity fleet status) can only measure
fork-vs-CHI capability once forks declare capabilities uniformly, which Conversion 2's generated
config provides. Do not start 3 before 2's `--check` gate is green in both forks.

## Conversion 1 — Engine distribution as a published, hash-verified artifact

### The decision (recorded)

The engine stops being distributed by humans applying diffs and becomes a **versioned release
artifact published from CHI**. Each fork pins an engine version + sha256 in a small lockfile. At
deploy time the fork downloads the pinned engine release, verifies the hash, splices the ENGINE
blocks into index.html between the existing fences, runs the static merge gate and the smoke test,
and only then uploads the Pages artifact. Parity becomes true by construction; the fence-based
parity checker is demoted from drift-detector to a post-assembly assertion. A repository_dispatch
fan-out from CHI opens a version-bump PR in each sibling on every new engine release, and that PR
is gated exactly like any other PR.

Why releases and not `actions/upload-artifact`: Actions artifacts auto-delete. Per GitHub Docs
("Configuring the retention period for GitHub Actions artifacts and logs"), "the artifacts and
log files generated by workflows are retained for 90 days before they are automatically deleted…
For public repositories: you can change this retention period to anywhere between 1 day or 90
days. For private and internal repositories: you can change this retention period to anywhere
between 1 day or 400 days." A distributed engine must be permanent, so it lives as a **GitHub
Release asset**, which does not expire. As of the GA on **28 October 2025** ("Immutable releases
are now generally available"), releases can be marked immutable — "Once you publish a release as
immutable, its assets can't be added, modified, or deleted… Immutable releases receive signed
attestations." We enable this on the CHI repo so a published engine version can never be silently
re-pointed.

Why deploy-time assembly and not runtime fetch: the user-facing product must stay one
hand-readable file served to users. Subresource Integrity (SRI) on a runtime `<script src>` would
keep the page honest but would split the app into two network fetches and break the "one
index.html" invariant. Assembly happens in CI; users still get a single document.

### New files

- `scripts/build_engine_artifact.py` (CHI only) — extracts the 45 index.html ENGINE blocks + 2
  sw.js ENGINE blocks by fence markers, concatenates them in a deterministic order into
  `engine.bundle.js`, writes `engine.manifest.json` (version, per-block byte lengths, ordered
  block-name list, sha256 of the bundle), and prints the sha256. Imports the block-extraction code
  from `scripts/check_engine_parity.py` — do not fork the parser.
- `scripts/apply_engine.py` (all forks) — reads `engine.lock.json`, downloads the pinned release
  assets, verifies sha256, splices each ENGINE block back into index.html and sw.js between its
  fences. Fails hard on hash mismatch, missing block, fence-count mismatch, or any block whose
  byte length disagrees with the manifest.
- `engine.lock.json` (all forks) —
  `{ "engine_version": "engine-v1.4.0", "sha256": "…", "source_repo": "ThursdaysFamous/DistrictExplorer-CHI" }`.
- `.github/workflows/release-engine.yml` (CHI) — on tag push `engine-v*`: runs
  build_engine_artifact.py, runs validate_index + smoke as a self-check, `gh release create` with
  the bundle + manifest as immutable release assets, then fans out repository_dispatch to every
  sibling.
- `.github/workflows/engine-bump.yml` (siblings) —
  `on: repository_dispatch: types: [engine-release]`: updates engine.lock.json to the new
  version+sha, runs apply_engine.py, runs validate_index + smoke, opens a PR on a fixed bot branch
  (force-push + `gh pr list` duplicate guard, same pattern as the roster workflows).

### Exact commands

Publish (CHI, after tagging `engine-v1.4.0`):
```
python scripts/build_engine_artifact.py --out dist/
gh release create engine-v1.4.0 dist/engine.bundle.js dist/engine.manifest.json \
  --title "Engine v1.4.0" --notes-file docs/engine-changelog/v1.4.0.md
```
`gh release create <tag> [files…]` auto-creates the git tag from the default branch HEAD if it
does not exist; add `--target <sha>` to pin the commit or `--verify-tag` to abort if the tag is
missing. Internally it creates a draft, uploads assets, then publishes — immutability is enforced
only after publish, so all assets are attached atomically.

Consume (sibling deploy step, before Pages upload):
```
gh release download "$(jq -r .engine_version engine.lock.json)" \
  --repo ThursdaysFamous/DistrictExplorer-CHI \
  --pattern 'engine.bundle.js' --pattern 'engine.manifest.json' --clobber
echo "$(jq -r .sha256 engine.lock.json)  engine.bundle.js" | sha256sum --check -
python scripts/apply_engine.py
python scripts/validate_index.py
node scripts/smoke_test.mjs
```
Note the TWO spaces between hash and filename in the `sha256sum --check` line — a single space or
CRLF line endings break parsing. `sha256sum --check` exits non-zero on any mismatch, which fails
the job. `gh` authenticates in Actions via `GH_TOKEN: ${{ github.token }}` for the download (CHI
is public, so read needs no extra scope).

### Token setup (least privilege)

The fan-out from CHI needs write on siblings; `GITHUB_TOKEN` is scoped to CHI only. Create a
**fine-grained PAT** (or a GitHub App installation token) scoped to ONLY the sibling repos, with
**Contents: read and write** and **Metadata: read** — the documented minimum to trigger a
repository_dispatch and open a PR. Store it as the secret `ENGINE_DISPATCH_TOKEN`. Do NOT use a
classic `repo`-scope PAT; it grants every repo on the account. A GitHub App is the better
long-term choice (short-lived installation token, auditable permission set), but a fine-grained
PAT scoped to named repos is acceptable for a two-person project. The repository_dispatch
client-payload is capped at 10 top-level properties (GitHub API returns "No more than 10
properties are allowed"); send `{ "engine_version": "…", "sha256": "…" }` and nothing else.
Reminder: a repository_dispatch only triggers a workflow that lives on the sibling's default
branch.

### Integrity approach (honest recommendation)

sha256 in engine.lock.json + manifest is the load-bearing check. On top of that, enable
**immutable releases** on the CHI repo so every published engine gets a GitHub-signed attestation
for free and the tag can never move; verify with `gh release verify <tag>`. That is proportionate.
**cosign/Sigstore key management is overkill here:** GitHub's immutable-release attestations
already use the Sigstore bundle format under the hood with ephemeral keys and no key custody.
`actions/attest-build-provenance` (GA June 2024) is available and cheap, but for a two-person
civic project whose forks verify a self-published sha256 against a pinned lockfile, SLSA
provenance adds ceremony without changing the threat model. Record this decision; do not add
cosign.

Dependency-bump automation: Dependabot cannot track a custom manifest like engine.lock.json, so
do not try. The repository_dispatch bump-PR automation above IS the version-bump mechanism.
Renovate's custom regex manager + custom datasource CAN watch a plain version file against an
HTTP/GitHub-releases datasource, and the Renovate GitHub App is free for open-source repos; name
it as an alternative if the dispatch fan-out ever proves flaky, but the dispatch model is
preferred because it is push (instant on release) rather than poll.

### Migration sequence from the fence system

1. Ship build_engine_artifact.py + apply_engine.py in CHI. Prove round-trip: extract → apply into
   a scratch copy of index.html → `git diff` is empty.
2. Publish engine-v(current) as the first release. Add engine.lock.json to CHI and both forks
   pinned to it.
3. Wire apply_engine.py into each fork's deploy-pages.yml BEFORE the rsync/_site assembly. Deploy
   once manually per fork; confirm chidistricts.com and nyc.chidistricts.com are byte-identical to
   pre-migration.
4. Demote parity: `scripts/check_engine_parity.py` stops being the weekly drift detector against
   deployed sites and becomes a **post-assembly assertion** inside the deploy job (assert the
   spliced ENGINE blocks equal the downloaded bundle). Keep engine-parity.yml running one more
   cycle as belt-and-suspenders, then delete it.
5. The fences STAY. They are now assembly markers, not review boundaries.

### Rollback

Pin the previous version + sha in engine.lock.json and redeploy. Because releases are immutable,
the old artifact is guaranteed still present and unchanged. No code revert needed.

### The drill (this is what makes the conversion DONE)

On a branch in NYC, corrupt the downloaded bundle (append a byte) after download but before verify,
or set a wrong sha in engine.lock.json. Push. CI MUST fail at the `sha256sum --check` step (or
apply_engine.py's hash gate). Record the failing run URL. Separately, confirm the first real
automated bump PR opens in NYC and passes validate_index + smoke.

- **Convention killed:** manual diff-porting sessions; the "port the diff, not the prompt" human
  N-1 port/deploy loop in ENGINE_SYNC.md (the principle survives as the artifact model's
  foundation; the human loop is gone).
- **Machine check that replaces it:** hash-verified deploy-time assembly + post-assembly parity
  assertion + gated bump-PR CI.
- **Prose demoted when:** the first automated engine-bump PR merges in NYC. From that date,
  ENGINE_SYNC.md's manual porting steps are struck through and marked superseded by this section.
- **CONVERSION 1 DONE — demotion fired 2026-07-13.** Drill evidence (metro-#3 gate input): the
  corruption drill failed CI at `sha256sum --check` in NYC —
  <https://github.com/ThursdaysFamous/DistrictExplorer-NYC/actions/runs/29209945390> (recorded
  with an invalid first attempt in NYC's `docs/NYC_CONFORMANCE.md`); unplanned second red, same
  gate family: the release workflow refused a mis-pointed `engine-v1.0.1` tag whose commit's
  lockfile still pinned v1.0.0 —
  <https://github.com/ThursdaysFamous/DistrictExplorer-CHI/actions/runs/29212754659>. The first
  automated bump PR (DistrictExplorer-NYC#23, from the engine-v1.0.1 fan-out) merged green the
  same day; ENGINE_SYNC.md's porting steps are struck through as of this date. Remaining
  Conversion 1 tail: delete Chicago's `engine-parity.yml` after one more weekly cycle
  (migration step 4).

## Conversion 2 — Per-fork docs/config generated from metro-worksheet.json

### The decision (recorded)

Every per-fork document and config block that a human currently hand-copies is instead GENERATED
from a single machine-readable `metro-worksheet.json` per fork. A generator emits the generated
regions; a `--check` mode regenerates to a temp dir, diffs against what is committed, and fails CI
on any drift. Hand-written and generated content coexist via `GENERATED:BEGIN`/`GENERATED:END`
fences, exactly like the engine's fences.

### Worksheet schema (metro-worksheet.json)

One file per fork, validated by JSON Schema before generation. Required keys:
- Identity: `this_metro`, `metro_name`, `metro_bbox`, `metro_center`, `permalink_gate`.
- Socrata: `socrata_host`, `socrata_app_token` (a secret reference, not a literal).
- Feedback/repo: `repo_issues`, `feedback_subject`.
- `metro_explorers`: list of `{id,label,url,emoji,bbox}` — generated INTO the config from the
  fleet manifest (Conversion 3), not authored here.
- Rendering: `highlight_class`, `poi_pin_class`, `hover_number_keys`, `hover_name_keys`.
- `layers`: list of `{id,label,group,area_rank}` — drives EXPECT_LAYER_IDS and LAYER_AREA_RANK.
- `anchors`: ≥3 API-free anchor layers, each with ground-truth `{lat,lng,expected}` points; plus
  one negative no-district point.
- `data_sources`: registry rows (Socrata id or shapefile provenance URL + vintage).
- `workflows`: inventory of this fork's scraper/builder workflows and schedules.
- `geocoder`: provider(s) for this metro.
- `palette`, `domains`: colors and canonical URLs.

### Generator

`scripts/generate_metro_files.py` — Python stdlib + `jsonschema` (pinned). Validates the worksheet
against `schema/metro-worksheet.schema.json`; a malformed worksheet fails before any file is
touched. Emits, into GENERATED regions only:
- `CLAUDE.md` — city, geocoder, workflow inventory, ground-truth point.
- `README.md` skeleton sections.
- The `METRO:BEGIN` block in index.html.
- The sw.js METRO config + CACHE_NAME.
- validate_index.py config constants (registerLayer floor, EXPECT_LAYER_IDS, LAYER_AREA_RANK,
  data floors).
- smoke_test.mjs constants (layer count, anchor points, expected classifications, negative point).

`--check` mode regenerates every target to a temp dir, diffs the GENERATED regions against the
committed files, exits non-zero on any difference. Wire it as a new CI step in each fork (or fold
into validate_index.py).

### Honest recommendation: bespoke generator, not cruft/cookiecutter

cruft (`cruft check` / `cruft update`, built on cookiecutter, actively maintained through 2025) is
real prior art that does exactly template-drift detection in CI — "`cruft check` can be added to CI
pipelines to ensure projects don't unintentionally drift" — and would work. But this project's
ethos is zero-dependency, no-framework, one-hand-readable. cruft pulls in cookiecutter + jinja2 +
a `.cruft.json` state file and a git-URL template-linkage model heavier than the problem. A
~200-line bespoke `generate_metro_files.py` with `--check` is more proportionate, easier for a
cheap agent thread to reason about, and keeps the toolchain to Python stdlib + jsonschema.
**Recommendation: build the bespoke generator. Name cruft explicitly as the escape hatch** — if
the number of generated files or forks grows past what a bespoke generator can maintain, migrate to
a cookiecutter template + cruft rather than growing the bespoke tool indefinitely.

### The drill

Regenerate NYC's CLAUDE.md from nyc/metro-worksheet.json; delete the stale hand-copied version.
In the same PR, replace any per-fork copy of a CHI-mastered playbook (docs/BUILD_PLAYBOOK_1.md,
docs/OPTIMIZATION_PLAYBOOK.md, docs/REDISTRICTING_RUNBOOK.md) with a pointer stub in the
METRO_EXPANSION_PLAYBOOK.md stub style — a fork may carry generated docs and pointer stubs, never
copies. Then hand-edit one generated region in a committed file (e.g. flip a ground-truth value)
and push. CI's `--check` step MUST fail. Record the failing run.

- **Convention killed:** hand-copied per-fork docs and config blocks, and per-fork copies of
  CHI-mastered playbooks.
- **Machine check that replaces it:** `generate_metro_files.py --check` drift gate in every fork's
  CI.
- **Prose demoted when:** NYC's CLAUDE.md is regenerated from its worksheet, the old stale copy
  is deleted at HEAD, and NYC's duplicated playbooks are pointer stubs. From that commit,
  hand-editing generated regions is a CI failure, not a review nit.

## Conversion 3 — Back-port gating (reverse-parity) via fleet manifest + fleet-status workflow

### The decision (recorded)

Fork-born engine-quality improvements must land in CHI within one release cycle, made mechanical.
Two pieces:

1. **metros.json on the canonical domain** (chidistricts.com/metros.json) — the single source of
   the fleet's `metro_explorers` list, solving the "edit N+1 repos to launch metro N+1" problem.
   Runtime behavior is deliberately UNCHANGED: the app does NOT fetch metros.json at runtime. It is
   consumed at GENERATION time (Conversion 2) to emit each fork's METRO_EXPLORERS block, with the
   hardcoded copy remaining the offline fallback exactly as today. This is the conservative choice:
   no new runtime network dependency, no new per-layer failure surface, and the "one manifest"
   benefit is realized where it matters (provisioning), not where it adds risk (page load).
2. **A weekly fleet-status workflow in CHI** aggregating, per fork: deployed engine version vs
   latest CHI release; validator capability comparison (each fork's validate_index.py declares a
   `CAPABILITIES` list of strings; the CHI job diffs them — any capability present in a fork but
   absent in CHI is a **reverse-parity WARN**); scraper last-green + per-field coverage rollup;
   open roster-PR count. Output: one auto-updated tracking issue or a committed STATUS.md.

### Definition-of-Done addition to ENGINE_SYNC.md

A fork-side engine-quality improvement's PR is **not done until the CHI release containing it is
tagged.** The fork PR description must link the CHI tracking issue; the fork PR may merge, but the
improvement's line item in fleet-status stays WARN until the CHI release ships.

### The drill

Introduce a capability in NYC's validate_index.py CAPABILITIES that CHI lacks (this already exists
in reality: EXPECT_LAYER_IDS, LAYER_AREA_RANK). Run fleet-status. It MUST emit a reverse-parity
WARN. Then ship the back-port in a CHI release; the next fleet-status run MUST clear it.

- **Convention killed:** discretionary back-porting.
- **Machine check that replaces it:** reverse-parity WARN in the weekly fleet-status workflow.
- **Prose demoted when:** the first reverse-parity WARN fires and is cleared by a CHI release. The
  "reference implementation" is now the union of fleet capabilities enforced by the WARN, not the
  seniority of code.
- **CONVERSION 3 DONE — demotion fired 2026-07-13.** The drill ran exactly as specified, with a
  REAL fork-born capability: NYC's `negative-point-ground-truth` validator check (born in
  DistrictExplorer-NYC#28 — where it immediately caught the mid-East-River negative point sitting
  inside the county-derived judicial-district geometry). Fleet-status run
  <https://github.com/ThursdaysFamous/DistrictExplorer-CHI/actions/runs/29217295970> fired the
  reverse-parity WARN on the tracking issue (#98); the back-port (CHI #99) shipped in the
  engine-v1.0.3 release per the ENGINE_SYNC definition of done; the next run
  <https://github.com/ThursdaysFamous/DistrictExplorer-CHI/actions/runs/29217657897> cleared it.
  metros.json is live at chidistricts.com/metros.json; `--sync-fleet` distributes the fleet list
  at generation time; runtime is untouched, as decided.

## Closing

**Execution order:** 1 → 2 → 3, for the reasons in the preamble. Engine artifact first (unblocks
and de-risks everything), doc generation second (needs the worksheet the artifact step reads),
reverse-parity last (needs uniform CAPABILITIES declarations from step 2).

**Deliberately NOT mechanized:**
- Roster PR human review stays. Officeholder changes are always human-reviewed via PR on a fixed
  bot branch; no auto-commit to main. This is a safety property, not a convention to optimize away.
- The honesty rules (never-guess/never-stale officeholder data, sanitized external strings,
  per-layer failure isolation) stay as prose in CLAUDE.md and are enforced by the smoke test's
  failure-isolation assertion, not by a new machine.
- Dual-map / transition-period support (see REDISTRICTING_RUNBOOK.md) is out of scope.

**The metro-#3 gate:** no new metro is provisioned until all three conversions' checks have EACH
failed once (the three drills above, with recorded failing-run URLs). Machinery you have never seen
fail is machinery you do not know works.

**Self-archival:** when all three "prose demoted when" conditions have fired, move this file to
docs/archive/ with a one-line header noting the completion date. A completed conversion schedule
left at docs/ is the anti-pattern named in the preamble — this document must not outlive its own
demotions.
